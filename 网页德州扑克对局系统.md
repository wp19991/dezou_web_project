# 网页德州扑克对局系统开发文档（完善版）

## 1. 文档定位

本文档用于指导一个**可实际落地开发、联调、测试、验收**的网页德州扑克对局系统实现。  
在保留原始方案“Python + FastAPI + pokerkit + 模板前端”的基础上，补齐以下内容：

- 明确游戏规则边界与实现范围
- 消除“以及其它可能参数/字段”这类不确定表述
- 细化状态机、接口契约、错误码、事件模型
- 增加数据持久化、回放、一致性、并发处理方案
- 给出可直接执行的工程目录、测试策略与里程碑

## 2. 原稿审查结论

原稿整体方向是合理的，已经具备以下正确骨架：

- 目标明确：单人操控、多 seat、后端规则为真相源、前端展示所有手牌
- 分层方向正确：规则内核、状态推进、聊天、事件流、Web UI
- API 主线完整：create / start-hand / state / act / chat / events / hands / replay
- 测试和里程碑有基本轮廓

但原稿存在几个会直接影响开发落地的问题：

1. **核心规则边界未定**
   - 未明确是无限注德州（No-Limit Hold'em）还是别的变体
   - 未定义 heads-up（2 人）时按钮位/盲注位/行动顺序
   - 未定义 all-in、边池、平分底池等规则
   - 未定义手牌何时自动结束、何时进入摊牌

2. **接口契约存在歧义**
   - 多处出现“以及其它可能的参数/数据模型”，实现时无法对齐
   - `act` 接口对非法请求写成“忽略”，不利于前端联调；应返回明确错误码
   - `raise` / `all-in` 的 `amount` 语义未定义：是总下注额、增量，还是本次补入额
   - 未定义接口错误格式、状态码、幂等处理

3. **状态与持久化不完整**
   - Session、Hand、Event 字段不够支撑回放和审计
   - 未定义 SQLite 表结构与最小落库范围
   - 未定义事件流与快照的关系，回放如何重建状态不清晰

4. **工程级缺失**
   - 未定义并发控制（同一 session 多次点击 act）
   - 未定义日志、配置、健康检查、异常处理
   - 未定义前端轮询策略、错误提示、加载状态
   - 未定义测试覆盖矩阵与验收脚本内容

## 3. 项目目标与范围

### 3.1 项目目标

实现一个基于网页的德州扑克对局系统，满足以下要求：

- 后端使用 `FastAPI` 提供牌局控制 API
- 使用 `pokerkit` 作为规则计算核心封装库
- 前端使用原生 `HTML + CSS + JavaScript` 模板页
- 支持 `2-9` 名玩家同时参与一桌
- 采用**单操作者模式**：同一个用户可代替任意 seat 提交动作
- 前端展示所有玩家手牌，便于调试、教学、演示
- 后端负责牌局规则校验、状态推进、结算与历史记录
- 支持聊天、事件流、历史列表和单手回放
- 为未来接入 AI/Agent 玩家预留标准 seat actor 接口，但本期不实现自动决策

### 3.2 非目标

本期不包含以下能力：

- 不做真人多账号在线系统
- 不做注册、登录、权限系统
- 不做锦标赛功能（盲注增长、报名、淘汰赛程）
- 不做复杂保险、straddle、run it twice 等扩展规则
- 不做头像、好友、房间大厅等社交能力
- 不实现 AI 自动打牌，仅预留扩展接口
- 不实现移动端适配为主目标，优先桌面浏览器体验

## 4. 规则边界与业务约定

### 4.1 游戏类型

本项目固定实现：

- **No-Limit Texas Hold'em（无限注德州扑克）**
- 52 张标准扑克牌，无大小王
- 每位玩家 2 张底牌
- 公共牌按 `flop / turn / river` 发出
- 支持 2-9 人同桌

### 4.2 房间模式

- 单桌单局会话模型
- 一个 `session` 表示一张牌桌
- 一个 `hand` 表示一手牌
- 同一 session 可以连续开始多手牌
- 本项目默认是**现金桌风格**：
  - 玩家初始筹码在创建 session 时确定
  - 手牌结束后，筹码带入下一手
  - 本期**不支持**中途加码（rebuy / add-on）
  - 本期**不支持**玩家中途离桌/入座；seat_count 在 session 创建后固定

### 4.3 盲注与庄位规则

- 创建 session 时必须指定：
  - `small_blind`
  - `big_blind`
  - `starting_stack`
- 约束：
  - `small_blind > 0`
  - `big_blind > small_blind`
  - `starting_stack >= big_blind * 20`（建议约束，避免无意义短码局）
- 每手牌需要确定庄位 `dealer_seat`
- 非 heads-up 时：
  - 小盲 = 庄位顺时针下一位
  - 大盲 = 小盲顺时针下一位
  - preflop 第一行动者 = 大盲顺时针下一位
  - postflop 第一行动者 = 庄位顺时针下一位的在手玩家
- heads-up（2 人）时特殊规则：
  - 庄位同时为小盲位
  - 另一位为大盲位
  - preflop 先行动的是小盲（也就是庄家）
  - flop/turn/river 先行动的是大盲

### 4.4 动作规则

本项目支持动作：

- `fold`
- `check`
- `call`
- `bet`
- `raise`
- `all_in`

说明：

- `bet`：当前轮无人下注时的主动下注
- `raise`：当前轮已有下注时的加注
- `all_in`：把剩余全部可用筹码投入
- 前端可以统一显示为“下注/加注”，但后端内部必须区分 `bet` 与 `raise`
- 为减少前后端歧义，`amount` 一律定义为：
  - **该玩家在当前下注轮结束后，本 seat 的“总投入额”**
  - 不是增量
  - 例如当前轮 seat_3 已投入 100，想加到 300，则 `amount = 300`
- `all_in` 时：
  - 允许 `amount` 为空，后端自动取该 seat 当前剩余可用筹码换算成该轮总投入额
  - 也允许前端传 `amount`，但后端会校验并归一化

### 4.5 合法动作生成原则

后端在任意可行动时点，必须返回：

- 当前行动 seat
- 当前轮 to_call 金额
- 当前最小 bet / 最小 raise_to
- 当前允许动作列表
- 每个动作所需参数范围

约束原则：

- 非当前行动者不能行动
- 已弃牌 seat 不能行动
- 已全下 seat 不能行动
- `check` 仅在 `to_call == 0` 时允许
- `call` 仅在 `to_call > 0` 时允许
- `bet` 仅在本轮尚无人下注时允许
- `raise` 仅在本轮已有下注且玩家剩余筹码足够时允许
- `all_in` 在 seat 仍在手且剩余筹码 > 0 时允许
- 所有动作必须由后端判定合法性，前端仅作预检查

### 4.6 全下、边池与摊牌

本项目必须支持：

- 一人或多人全下
- 主池与边池（side pot）
- 多名玩家摊牌比较牌力
- 平局分池

底池分配约定：

- 按池逐层结算
- 平分时若出现无法整除的奇数筹码，按**从庄位左侧开始顺时针**的顺序分配余数筹码，确保结果可复现

### 4.7 手牌结束条件

出现以下任一条件，本手牌结束：

1. 仅剩 1 名未弃牌玩家  
   - 直接获胜，不进入摊牌
2. 所有未弃牌玩家都已全下，且剩余公共牌尚未发完  
   - 系统自动补发剩余公共牌并进入结算
3. river 下注轮结束，仍有 2 名及以上未弃牌玩家  
   - 进入摊牌与结算

### 4.8 可见性规则

本项目采用**全明牌调试模式**：

- 所有 seat 的 hole cards 在前端全部可见
- 回放中也默认全部可见
- 后续若扩展真实对战模式，可增加视角过滤层，但本期不做

## 6. 总体架构

## 6.1 逻辑分层

1. **API 层**
   - FastAPI 路由
   - 参数校验
   - 返回统一错误格式

2. **应用服务层**
   - session / hand 生命周期编排
   - act / chat / replay 服务
   - 并发锁控制

3. **规则内核层**
   - `TableKernel`
   - 对 pokerkit 的封装
   - 合法动作生成、街道推进、结算

4. **持久化层**
   - session、hand、event、snapshot 落库
   - 历史查询与回放查询

5. **前端展示层**
   - 桌面 UI
   - 动作面板
   - 聊天面板
   - 时间线与历史列表

### 6.2 核心模块

#### `TableKernel`
职责：

- 初始化一手牌
- 维护牌桌规则对象
- 计算当前行动上下文
- 应用动作
- 自动推进街道
- 结算底池和胜者
- 生成 hand snapshot

#### `TurnCoordinator`
职责：

- 管理 session 的 phase
- 维护当前 hand 的生命周期
- 在动作提交后驱动自动推进
- 判断是否进入 `waiting_actor_action` 或 `hand_ended`

#### `SeatActor`
未来扩展接口，当前只有 `ManualSeatActor` 实现。

建议接口：

```python
class SeatActor(Protocol):
    def get_action(self, view: "ActorView", options: list["AvailableAction"]) -> "ActionCommand | None":
        ...
```

本期不由 seat actor 自动调用，保留接口即可。

#### `ChatService`
职责：

- 接收聊天消息
- 写入事件流
- 不干扰动作处理

#### `TraceStore`
职责：

- 追加事件
- 存储 hand 快照
- 支持按游标拉取事件
- 支持 replay

## 7. 状态机设计

### 7.1 SessionPhase

- `waiting_start`：session 已创建，未开始当前手
- `running`：系统内部推进中，不对前端暴露可行动作提交窗口
- `waiting_actor_action`：等待当前行动者提交动作
- `hand_ended`：本手结束，可查看结果并开启下一手
- `closed`：会话关闭（预留，本期可不开放 API）

### 7.2 状态流转

```text
waiting_start
  -> start-hand
running
  -> waiting_actor_action      (推进到某位玩家需要操作)
running
  -> hand_ended               (自动结算结束)
waiting_actor_action
  -> running                  (收到合法动作后重新推进)
hand_ended
  -> start-hand               (开始下一手)
```

### 7.3 设计原则

- `running` 是后端内部短暂状态
- 前端通常只会看到：
  - `waiting_start`
  - `waiting_actor_action`
  - `hand_ended`
- 任何动作请求在 `phase != waiting_actor_action` 时都应拒绝

## 10. 接口设计

统一前缀：`/api/v1`

说明：

- 原稿中的 `/core/...` 可以继续使用
- 但从工程规范看，推荐统一为 `/api/v1/...`
- 若你希望兼容原稿，建议做一层路由别名：
  - `/core/*` -> Deprecated Alias
  - `/api/v1/*` -> 正式路径

下文以 `/api/v1` 为正式路径说明。

### 10.1 通用响应格式

成功响应：

```json
{
  "ok": true,
  "data": {}
}
```

失败响应：

```json
{
  "ok": false,
  "error": {
    "code": "ACTOR_TURN_MISMATCH",
    "message": "当前不是该玩家行动回合",
    "details": {
      "expected_actor_id": "seat_3",
      "actual_actor_id": "seat_1"
    }
  }
}
```

### 10.2 错误码定义

| 错误码 | HTTP 状态码 | 说明 |
|---|---|---|
| `SESSION_NOT_FOUND` | 404 | session 不存在 |
| `HAND_NOT_FOUND` | 404 | hand 不存在 |
| `INVALID_REQUEST` | 422 | 参数校验失败 |
| `SESSION_PHASE_INVALID` | 409 | 当前阶段不允许该操作 |
| `HAND_ALREADY_RUNNING` | 409 | 当前已有未结束手牌 |
| `ACTOR_TURN_MISMATCH` | 409 | 不是当前行动者 |
| `ACTION_NOT_ALLOWED` | 409 | 动作不合法 |
| `AMOUNT_OUT_OF_RANGE` | 409 | 金额超界 |
| `SEAT_NOT_FOUND` | 404 | seat 不存在 |
| `REPLAY_NOT_READY` | 409 | 回放尚未生成 |
| `INTERNAL_ERROR` | 500 | 服务器内部错误 |

### 10.3 幂等与重复提交

建议在以下写接口支持可选字段 `request_id`：

- `POST /sessions`
- `POST /sessions/{session_id}/actions`
- `POST /sessions/{session_id}/chat`

规则：

- 同一 session 内，若短时间重复提交相同 `request_id`，返回首次结果
- 便于前端重复点击、网络重试时避免重复入账

## 11. API 详细定义

### 11.1 创建会话

`POST /api/v1/sessions`

请求体：

```json
{
  "session_id": "table-demo-001",
  "seat_count": 6,
  "small_blind": 50,
  "big_blind": 100,
  "starting_stack": 10000,
  "seat_names": ["玩家1", "玩家2", "玩家3", "玩家4", "玩家5", "玩家6"],
  "request_id": "req-create-001"
```

### 12.1 系统事件

- `session_created`
- `hand_started`
- `street_changed`
- `waiting_actor_action`
- `board_dealt`
- `showdown_started`
- `hand_ended`

### 12.2 动作事件

- `blind_posted`
- `action_applied`
- `folded`
- `checked`
- `called`
- `bet_to`
- `raised_to`
- `all_in`
- `pot_awarded`

### 12.3 聊天事件

- `chat_sent`

事件设计原则：

- 前端时间线与回放都从 event stream 读取
- 回放页面可以以 `events + final_state` 混合渲染
- 事件必须**只追加不修改**

## 13. 并发、一致性与事务处理

### 13.1 同一 session 串行写

对以下操作加 session 级互斥锁：

- start-hand
- actions
- chat（可选不加锁；但建议事件写入统一排队）

建议实现：

- 进程内：`asyncio.Lock` 按 `session_id` 维护
- 若未来多实例部署：需升级为外部锁（Redis 等），本期不做

### 13.2 事务边界

一次合法动作应在一个事务中完成：

1. 校验当前 phase / actor
2. 应用动作
3. 自动推进街道
4. 更新 state / hand / session
5. 追加事件
6. 若 hand 结束，写 replay snapshot

任何一步失败都应回滚，避免：
- 动作已执行但事件未写
- hand 已结束但 stacks 未更新

### 13.3 重复点击处理

- 前端按钮点击后立即 disabled
- 后端支持 `request_id`
- 相同 `request_id` 只处理一次

## 14. 前端交互设计

### 14.1 页面分区

1. **顶部控制区**
   - 创建 session
   - 开始新一手
   - 显示 session_id / hand_no / phase

2. **牌桌区域**
   - 庄位标记（D / SB / BB）
   - 各 seat 名称、筹码、当街投入、是否弃牌/全下
   - 当前行动者高亮
   - 所有 seat 的 hole cards

3. **公共牌区域**
   - flop / turn / river
   - 底池总额

4. **动作面板**
   - 当前行动 seat 展示
   - `弃牌 / 过牌 / 跟注 / 下注 / 加注 / 全下`
   - 金额输入框
   - 合法动作提示

5. **聊天面板**
   - 选择 speaker
   - 输入文字并发送

6. **时间线区域**
   - 按事件顺序展示
   - 区分 system/action/chat 样式

### 14.2 前端状态同步策略

本期推荐简单方案：

- 用户进入页面后先拉取 `state`
- 页面每 `1000ms` 轮询 `events`
- 每次收到 action/system 事件后，重新拉取一次 `state`
- `chat` 事件可仅增量插入时间线
- 不引入 WebSocket / SSE，本期以轮询为准

### 14.3 前端交互约束

- 只有当前行动者 seat 被选中时，动作按钮才可用
- 按钮文本根据 `available_actions` 动态显隐
- `bet/raise` 金额输入框根据 `min/max/default` 自动更新
- 非法响应必须以中文 toast / banner 展示
- 加载中必须禁用提交按钮，防重复点击

### 14.4 UI 文案建议

- `轮到 玩家4 行动`
- `当前需跟注：100`
- `最小加注到：300`
- `当前不是该玩家行动回合`
- `当前动作不合法`
- `金额超出允许范围`
- `本手已结束，请开始下一手`

## 22. 开发约束与关键决策汇总

为避免实现分歧，以下结论固定：

1. 游戏类型固定为 **No-Limit Texas Hold'em**
2. `amount` 语义固定为 **当前街总投入额**
3. 非当前行动者请求 **返回错误，不静默忽略**
4. 本期支持 **all-in 与 side pot**
5. 本期采用 **全明牌调试模式**
6. 本期前端使用 **轮询，不做 WebSocket**
7. 本期采用 **单桌单会话**，不做多房间大厅
8. session 写操作必须 **串行化**
9. replay 使用 **事件流 + 最终快照** 组合实现
10. `hand_no` 由后端生成，前端不指定

## 24. 最终建议

如果你准备按这版文档实施，我建议第一版开发时坚持两个原则：

- **先让规则与状态机完全闭环，再做 UI 美化**
- **所有关键行为都先写事件，再写回放，再做前端展示**

这样可以最大程度避免：
- 前端能点但状态错乱
- 规则跑通但回放做不出来
- 聊天和动作交叉后难以排查问题
